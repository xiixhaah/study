## 一、JWT 核心知识体系

### 1.1 JWT 核心概念

JWT 全称 **JSON Web Token**，是基于 RFC 7519 标准的开放数据交换规范，核心是在网络环境中，以紧凑、自包含、防篡改的方式，在前后端之间安全传输JSON格式的信息。
核心特点：无状态、自包含、不可篡改，最核心的应用场景是前后端分离架构的用户身份鉴权，也可用于多方系统的安全信息交换。

### 1.2 JWT 的三大组成部分

JWT 由三部分组成，三部分通过`.`拼接成完整的Token字符串，格式为`Header.Payload.Signature`。

1.  **Header（头部）**
    是一个JSON对象，核心包含两个字段：`alg`（签名算法，常用HS256、RS256）、`typ`（令牌类型，固定为JWT）。最终会通过Base64Url编码，形成JWT的第一部分。
    注意：Base64Url是编码而非加密，可反向解码，不可存放敏感信息。
2.  **Payload（载荷）**
    是JWT的核心数据载体，JSON格式，存放需要传输的声明（Claims），分为三类：
    - 注册声明：标准推荐字段，如`iss`（签发者）、`exp`（过期时间）、`sub`（主题）、`aud`（受众）等；
    - 公共声明：可自定义的通用字段；
    - 私有声明：业务自定义字段，如用户ID、角色、权限等。
      最终同样通过Base64Url编码形成第二部分，**同样不是加密，绝对不能存放密码等敏感信息**。
3.  **Signature（签名）**
    是JWT防篡改的核心，生成逻辑为：将编码后的Header + 编码后的Payload，使用Header中指定的算法，加上服务端独有的秘钥（secret）进行签名生成。
    只有服务端持有秘钥，前端无法篡改，服务端通过校验签名，即可确认Token是否被篡改、是否合法。

### 1.3 JWT 示例

完整JWT示例：`eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOjEwMDEsInJvbGVzIjpbImFkbWluIl0sIm5hbWUiOiJKb2huIiwiZXhwIjoxNzE0MDAwMDAwfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c`

- 第一部分：Header编码结果，对应原始JSON `{"alg":"HS256","typ":"JWT"}`
- 第二部分：Payload编码结果，对应原始JSON `{"userId":1001,"roles":["admin"],"name":"John","exp":1714000000}`
- 第三部分：HS256算法+服务端秘钥生成的签名

### 1.4 JWT 完整鉴权流程

1.  用户提交账号密码，发起登录请求到服务端；
2.  服务端校验账号密码合法性，校验通过后，根据用户信息生成JWT，返回给前端；
3.  前端收到JWT后，存储在本地（localStorage/sessionStorage/cookie）；
4.  前端后续发起业务请求时，在请求头（标准格式`Authorization: Bearer <JWT>`）中携带JWT；
5.  服务端收到请求后，先解析JWT，用秘钥重新计算签名，与Token中的签名比对，同时校验Token是否过期、签发者是否合法；
6.  校验通过，执行业务逻辑并返回数据；校验不通过（篡改、过期、无效），直接返回401/403错误，拒绝访问。

### 1.5 补充：JSON与对象的区别及转换

#### 核心区别

一、核心本质
JSON 是符合规范的纯文本字符串格式，核心用于跨端、跨语言数据交换；JS 原生对象是JavaScript 运行在内存中的键值对数据结构，核心用于代码内部的数据处理与逻辑运算。
二、四大核心区别
1、本质属性不同
JSON：轻量级通用数据交换格式，本质是标准化纯字符串，不依附于任何编程语言。
JS 对象：JavaScript 语言内置的运行时数据结构，是内存中的键值对集合，仅在 JS 语言环境中生效。
2、语法规则不同
JSON 语法严格受限：键名必须用双引号包裹，单引号、无引号均不合法；仅支持 6 种数据类型：字符串、数字、布尔值、null、数组、合规对象；禁止使用函数、undefined、Symbol、循环引用，不允许出现尾逗号、代码注释。
JS 对象语法灵活宽松：键名无引号、单引号、双引号均可正常使用；支持 JS 全量数据类型，可包含函数方法、getter/setter；支持尾逗号、代码注释、循环引用。
3、核心用途不同
JSON：专属跨语言、跨端、跨网络的数据传输，是前后端交互、数据持久化存储的通用标准格式。
JS 对象：专属 JS 语言内部的数据处理、逻辑运算、程序状态存储，服务于业务代码的逻辑实现。
4、传输与存储特性不同
JSON：纯字符串格式，可直接在网络中传输、写入文件持久化存储。
JS 对象：内存中的运行时数据，无法直接网络传输 / 持久化，必须先序列化转为 JSON 字符串后才可操作。

#### 相互转换方式（JS环境）

1.  **对象 → JSON**：使用`JSON.stringify()`方法，将JS对象序列化为符合JSON规范的字符串，会自动忽略函数、undefined、Symbol类型属性；
2.  **JSON → 对象**：使用`JSON.parse()`方法，将合规的JSON字符串反序列化为JS原生对象，字符串不符合JSON规范时会抛出解析错误。

### 1.6 双Token机制：access_token 与 refresh_token

#### 核心定义

- **access_token（访问令牌）**：业务接口鉴权的核心凭证，就是上述的JWT，特点是**有效期极短**（通常30分钟-2小时），每次业务请求都需要携带，是访问资源的直接凭证。
- **refresh_token（刷新令牌）**：专门用于刷新获取新的access_token，特点是**有效期长**（通常7天-30天），不参与普通业务接口的鉴权，仅在access_token过期时，调用专属的刷新接口使用，安全性要求更高，可采用JWT格式或服务端存储的随机字符串。

#### 双Token机制解决JWT的核心痛点

单Token模式下JWT存在3个致命问题，双Token可完美解决：

1.  **解决「JWT无法主动作废，安全风险高」的问题**
    JWT是无状态的，一旦签发，过期前服务端无法主动失效。若用户改密、账号被盗、权限变更，只要Token未过期，攻击者依然可访问系统。
    双Token解决方案：access_token有效期极短，就算泄露，可用窗口极短，风险大幅降低；refresh_token仅用于刷新接口，暴露概率极低，同时可在服务端做状态管理（存入Redis/数据库），用户改密、登出、账号异常时，服务端可直接拉黑refresh_token，彻底阻断攻击者获取新的access_token，解决JWT无法主动失效的核心问题。

2.  **解决「长有效期风险高，短有效期体验差」的矛盾**
    单Token模式下，有效期太长被盗风险高，太短则用户需要频繁登录，体验极差。
    双Token解决方案：短有效期的access_token保证安全，长有效期的refresh_token可在access_token过期时，前端无感调用刷新接口获取新Token，用户无需重新登录，完美平衡安全性与用户体验。

3.  **解决「权限变更无法实时生效」的问题**
    单Token模式下，用户的角色、权限存在Payload中，Token签发后，就算服务端修改了权限，只要Token未过期，用户依然使用旧权限，存在越权风险。
    双Token解决方案：每次用refresh_token刷新access_token时，服务端都会重新拉取用户最新的角色、权限，生成新的Payload写入新Token。用户权限变更后，最多在一个access_token有效期内即可同步最新权限，不影响用户体验。

## 二、如何抽离逻辑为Hook

### 2.1 核心概念

Hook抽离的本质，是**将组件中与UI无关的、可复用的业务逻辑、状态逻辑、副作用逻辑，封装成独立、可组合、高内聚低耦合的函数**，实现逻辑复用、与UI解耦，让组件仅负责UI渲染，同时提升代码的可维护性、可测试性。

### 2.2 抽离Hook的核心原则

1.  **单一职责原则**：一个Hook只负责一件事，只处理一个独立的逻辑单元，不把多个不相关的逻辑耦合在一个Hook中；
2.  **高内聚低耦合**：相关的状态、方法、副作用全部内聚在Hook内部，对外仅暴露必要的状态和方法，外部依赖通过参数传入，不强绑定组件的props/状态，保证通用性；
3.  **无UI依赖**：Hook仅负责逻辑，绝对不包含UI渲染、DOM操作、样式相关代码，保证可在任意符合条件的组件中复用；
4.  **可组合性**：Hook之间可互相调用、组合，复杂逻辑通过多个简单Hook组合实现，而非写超大体积的Hook；
5.  **副作用可控**：Hook内部的副作用（接口请求、定时器、事件监听）必须做好生命周期管理，组件卸载时清除副作用，避免内存泄漏，同时通过依赖项严格控制副作用触发时机。

### 2.3 抽离Hook的通用落地步骤

1.  **识别可抽离逻辑**：在组件中找到与UI渲染无关、重复出现、逻辑复杂度高的代码块，比如分页逻辑、表单校验、接口请求的loading/错误处理、权限判断、弹窗显隐逻辑等；
2.  **剥离逻辑与UI**：将这部分逻辑的状态、计算属性、方法、副作用，全部从组件中剥离，放入独立的Hook函数中；
3.  **定义入参与出参**：入参为Hook运行需要的外部依赖（如接口参数、配置项），出参为组件需要用到的状态和方法，仅暴露必要内容，隐藏内部实现细节；
4.  **完善生命周期管理**：处理好Hook内部的副作用，比如请求取消、定时器清除、事件解绑，保证组件卸载时正确清理；
5.  **通用化优化**：将硬编码内容改为可配置参数，适配不同业务场景，拆分细粒度Hook，通过组合实现复杂功能；
6.  **补充单元测试**：Hook与UI解耦后极易测试，补充单测保证逻辑稳定性，重构时不会引入bug。

## 三、Axios封装与错误统一处理

### 3.1 Axios封装的核心目标

基于Axios的请求/响应拦截器，对网络请求做标准化统一处理，封装通用逻辑，减少业务代码重复开发，核心实现：统一请求配置、统一鉴权处理、统一数据格式化、**统一错误拦截与处理**，同时拓展请求防抖、重复请求取消、接口重试等通用能力。

### 3.2 Axios封装的核心实现逻辑

1.  **创建实例与全局基础配置**：基于`axios.create`创建独立实例，配置全局baseURL（区分开发/生产环境）、超时时间、跨域配置、默认请求头等，避免每个请求重复配置；
2.  **请求拦截器封装**：请求发送到服务端前，统一处理：自动携带Token、请求参数格式化、重复请求取消、全局loading管理等；
3.  **响应拦截器封装**：请求返回后，统一处理：成功响应的数据解构、业务状态码判断，以及核心的**错误统一分类与处理**。

### 3.3 错误统一处理（错误码体系）

将请求错误分为两大类，建立全场景的分类处理规则，避免业务代码重复写错误逻辑。

#### 第一类：HTTP协议层错误（HTTP状态码非2xx）

基于HTTP标准状态码，分区间做统一处理：

1.  **4xx 客户端错误**
    - 401（未授权）：Token过期/无效，双Token模式下先尝试刷新Token，刷新成功则重发原请求，刷新失败则清除本地Token，跳转登录页，提示用户登录过期；
    - 403（权限不足）：提示用户「暂无权限执行该操作」，不跳转登录，可跳转无权限兜底页；
    - 404（资源不存在）：提示用户「请求的资源不存在」，同时上报错误日志，排查接口地址问题；
    - 400（参数错误）：提取后端返回的错误信息，提示用户「参数错误：xxx」；
    - 429（请求限流）：提示用户「操作过于频繁，请稍后再试」，可配置限流重试逻辑。
2.  **5xx 服务端错误**
    - 500（服务端内部错误）：统一提示用户「服务器开小差了，请稍后再试」，不向用户暴露服务端细节，同时上报完整错误日志到监控平台；
    - 502/503/504：统一提示用户「网络/服务异常，请稍后再试」，超时场景可配置自动重试。
3.  **网络层非HTTP错误**
    - 无网络/网络断开：提示用户「网络异常，请检查网络连接」；
    - 请求超时：提示用户「请求超时，请稍后再试」；
    - 跨域错误：上报日志排查配置，生产环境做兜底提示。

#### 第二类：业务层错误（HTTP状态码200，业务状态码非成功值）

接口请求通了，但业务逻辑执行失败，需与后端约定统一的业务码规范，做分层处理：

1.  **约定全局业务码规范**：比如`code=0`代表成功，非0代表失败；分区间定义错误类型，1xxx为通用业务错误，2xxx为用户相关错误，3xxx为权限相关错误等；
2.  **通用处理逻辑**：通用业务错误直接提取后端返回的提示文案，通过全局消息组件统一提示，无需业务代码单独处理；
3.  **特定场景处理**：针对账号封禁、密码错误超限等特定业务码，做专属逻辑处理，比如跳转对应页面、锁定表单；
4.  **兜底与自定义配置**：无返回提示时用兜底文案「操作失败，请稍后再试」；同时支持配置跳过全局提示，交给业务代码自行处理特殊场景。

## 四、用户多权限体系处理方案

### 4.1 核心模型

企业级系统通用方案为**RBAC（基于角色的访问控制）模型**，核心逻辑是：**用户与角色挂钩，角色与权限关联**，通过角色作为中间层，解耦用户与权限，实现灵活的权限管控。

三层核心关联关系：

1.  **用户层**：系统登录用户，一个用户可绑定一个或多个角色；
2.  **角色层**：权限的集合载体，是中间层，一个角色可绑定多个权限点；
3.  **权限层**：系统最小权限单元，分为页面权限（路由权限）、操作权限（按钮/接口权限）两大类。

核心原则：前端做体验层的权限管控，后端做接口层的权限兜底校验，双重保障系统安全。

### 4.2 完整的权限处理流程

1.  后端搭建RBAC体系，建立用户表、角色表、权限表，以及用户-角色、角色-权限的关联表，实现多对多的权限管理；
2.  用户登录成功后，服务端根据用户ID，查询其绑定的所有角色，再通过角色查询对应的全量权限集合，返回给前端；
3.  前端将权限集合存入全局状态管理（Pinia/Vuex/Redux），并做持久化处理，避免刷新页面丢失；
4.  基于权限集合，做全链路的三层权限管控：
    - **路由权限（页面级）**：采用动态路由方案，系统初始化仅注册登录页等公共路由，用户拿到权限后，过滤出有权限的路由，动态添加到路由实例中；同时通过全局路由守卫做兜底拦截，无权限的路由访问直接跳转到403页；
    - **操作权限（按钮/元素级）**：封装全局权限指令/权限组件，传入所需权限码，自动判断用户是否拥有对应权限，无权限则隐藏/禁用对应元素，比如新增、编辑、删除按钮，无权限的用户不可见；
    - **接口权限（请求级）**：前端可在请求拦截器中做前置校验，拦截无权限的接口请求；核心安全兜底由后端实现，所有接口必须校验用户权限，无权限直接返回403，绝对不能仅靠前端做权限控制。

## 五、项目重构优化方案

### 核心原则

**先稳后优，渐进式重构**，在不改变现有业务功能、对外交互、用户体验的前提下，从宏观架构到微观代码分层次优化，核心目标是降低维护成本、提升可拓展性、建立质量防护网，而非全量推翻重写。

### 5.1 前置准备

1.  梳理项目架构、业务模块、技术债务，明确重构目标与优先级，优先优化维护成本最高的模块；
2.  搭建完整的开发、测试、灰度环境，保证重构代码可充分验证，支持一键回滚；
3.  统一团队编码规范、技术选型，避免重构后出现新的代码混乱。

### 5.2 宏观架构层重构优化

核心是解耦、分层、标准化，提升项目可拓展性。

1.  **组件架构拆分与重构**
    - 按职责拆分：将组件拆分为**纯UI展示组件**和**业务逻辑容器组件**。纯UI组件仅负责渲染，无业务逻辑、接口请求，数据与回调全通过props传入，可复用性极强；业务组件仅负责处理业务逻辑、状态管理、接口请求，两者完全解耦；
    - 按复用层级拆分：拆分为全局通用基础组件、业务通用组件、页面专属组件，消除重复组件，统一组件API设计，提升复用率。
2.  **接口层封装与标准化**
    - 基于Axios封装统一请求实例，实现统一拦截、错误处理、鉴权逻辑；
    - 按业务模块拆分API文件，所有接口的地址、方法、参数定义统一封装，业务组件仅需调用封装好的API方法，无需关心请求细节；
    - 统一接口入参、出参格式，消除业务代码中重复的接口处理逻辑。
3.  **状态管理重构与规范化**
    - 基于Pinia/Redux按业务模块拆分状态仓库，每个仓库遵循单一职责原则，仅负责对应模块的状态管理；
    - 分层管理状态：全局通用状态放入全局仓库，页面级状态放入对应页面仓库，组件内部状态留在组件自身，避免全局状态泛滥；
    - 将接口请求、数据格式化、缓存逻辑封装在仓库内部，保证全项目数据一致性，消除props层层传递的问题。
4.  **工程架构优化**
    - 按业务模块拆分路由配置，实现路由懒加载，统一路由守卫与权限逻辑；
    - 优化构建配置，制定分包策略、tree-shaking，提升构建速度与运行性能；
    - 接入ESLint、Prettier、Git Hooks，强制代码规范，避免不规范代码进入仓库。

### 5.3 微观代码层重构优化

核心是提升代码质量、可维护性、可测试性，清理技术债务。

1.  **单元测试体系建设**
    - 重构前先给核心逻辑、工具函数、Hook、组件补充单元测试，建立质量防护网；
    - 对修改的函数/组件，先跑通原有单测再修改，修改完成后再次验证，保证业务逻辑一致性；
    - 接入CI流程，提交代码自动跑单测，不通过无法合并。
2.  **函数与逻辑重构**
    - 拆分超长函数，遵循单一职责，一个函数只做一件事，提升代码可读性；
    - 抽离重复逻辑为通用工具函数、Hook，一处修改全项目生效，降低维护成本；
    - 消除硬编码，将魔法数字、固定字符串抽离为常量；用卫语句、策略模式替代深层if-else嵌套；用async/await优化异步逻辑，消除回调地狱。
3.  **冗余代码与技术债务清理**
    - 清理未使用的代码、依赖、组件、资源，删除死代码，减少项目体积；
    - 修复ESLint报错、TypeScript类型错误，完善类型定义，消除any类型，提升代码类型安全；
    - 分离纯计算逻辑与副作用逻辑，纯函数化改造，提升代码可测试性。

### 5.4 质量防护网建设

1.  测试防护网：单元测试、集成测试、E2E测试三层体系，核心功能全覆盖；
2.  规范防护网：代码规范、类型校验强制卡点，不规范代码无法提交合并；
3.  CI/CD防护网：自动化构建、测试、发布流程，支持灰度发布、一键回滚；
4.  监控防护网：接入前端监控平台，实时监控线上错误、性能异常，快速定位问题；
5.  文档体系：补充架构文档、组件文档、API文档、开发规范，降低团队协作与新人上手成本。

## 六、课程筛选功能常见问题与解决方案

### 6.1 核心问题1：切换筛选条件时页码未重置

#### 问题现象

用户在第3页切换分类、搜索关键词等筛选条件后，页码依然停留在第3页，导致新筛选条件下数据总量不足3页，页面显示空白、数据错乱。

#### 问题根源

筛选条件与分页参数是两个独立状态，切换筛选条件时仅更新了筛选条件，未同步重置页码为1，导致接口请求时携带旧页码，与新筛选条件不匹配。

#### 解决方案

1.  核心规则：**任意筛选条件的有效变更，必须同步将页码重置为1**，保证新筛选条件永远从第一页请求数据；
2.  细节优化：区分临时输入与确认变更，比如搜索关键词采用防抖处理，或等待用户点击搜索按钮/按下回车后，再重置页码发起请求，避免频繁接口调用；
3.  兜底校验：接口请求前做参数校验，若筛选条件与上一次请求不一致，强制将页码置为1；
4.  体验优化：重置页码的同时，将页面滚动到列表顶部，保证用户能看到最新筛选结果。

### 6.2 核心问题2：多分类筛选的联动与体验优化

#### 1. 父子分类联动错乱问题

- 问题：一级分类切换后，二级分类未同步更新，可选选项与当前一级分类不匹配；
- 解决方案：二级分类选项完全由选中的一级分类决定，一级分类切换时，清空已选中的二级分类，重新拉取/过滤对应二级分类选项，保证层级联动正确。

#### 2. 多条件组合请求频繁问题

- 解决方案：输入类筛选条件做防抖处理；多个筛选条件合并为一次确认操作后再发起请求，避免频繁接口调用；分类下拉选项做本地缓存，无需每次切换都重新拉取。

#### 3. 刷新页面筛选状态丢失问题

- 解决方案：将筛选条件、页码参数同步到URL的query参数中，页面刷新时从URL读取参数还原状态，同时支持链接分享，打开后直接还原筛选结果。

#### 4. 筛选结果为空的兜底处理

- 解决方案：展示友好的空状态提示，提供「清空筛选条件」的快捷按钮；对无数据的分类选项做置灰处理，避免用户无效选择。

## 七、课程列表首屏加载体验优化与性能指标

### 核心优化思路

从**网络层、资源层、渲染层**三个维度全链路优化，减少首屏加载耗时，提升用户体验，同时通过标准化性能指标衡量优化效果。

### 7.1 网络层优化

核心是减少请求耗时，提升资源加载速度。

1.  **协议升级**：优先使用HTTP/2协议，支持多路复用，解决HTTP/1.1队头阻塞问题，大幅提升多资源并行加载速度；可通过CDN厂商支持HTTP/3，进一步优化弱网环境表现；
2.  **CDN加速**：将静态资源（JS、CSS、课程封面图等）部署到CDN边缘节点，用户从最近的节点获取资源，降低网络延迟；配置合理的强缓存+协商缓存策略，避免重复加载资源；
3.  **预加载与预连接**：对首屏核心资源用`<link rel="preload">`提前加载，对接口/CDN域名用`<link rel="preconnect">`提前建立TCP连接，减少请求耗时；
4.  **接口请求优化**：合并列表相关的多个接口，减少请求次数；接口按需返回首屏必要字段，减少响应体体积；对不常变化的分类数据做本地缓存，减少无效请求。

### 7.2 资源层优化

核心是减少资源体积，提升加载效率。

1.  **代码分包与懒加载**：路由懒加载、非首屏组件懒加载，减少首屏主包体积；第三方大依赖单独分包，设置长期缓存，避免业务代码修改导致全量重新加载；
2.  **资源压缩优化**：构建时对JS、CSS做压缩、tree-shaking，删除未使用代码；开启gzip/brotli压缩，大幅减少传输体积；
3.  **图片资源优化**：课程列表的核心优化点，采用WebP/AVIF新一代图片格式，体积减少50%以上；实现图片懒加载，仅加载可视区域内的图片；响应式图片适配不同设备分辨率，避免加载超大图；无损压缩图片，减少体积。

### 7.3 渲染层优化

核心是减少浏览器渲染耗时，提升页面响应速度。

1.  **HTML解析与渲染优化**
    - CSS优化：首屏核心CSS内联到HTML头部，避免CSS加载阻塞渲染；非首屏CSS异步加载；优化CSS选择器，减少样式计算耗时；
    - JS加载优化：非关键JS脚本使用defer/async属性，避免阻塞HTML解析与渲染；defer等待HTML解析完成后按顺序执行，适合有依赖的业务脚本；async下载完成后立即执行，适合无依赖的第三方脚本；
    - SSR/SSG优化：采用Nuxt/Next实现服务端渲染，服务端返回渲染好的HTML，浏览器可直接渲染，大幅减少白屏时间；对不常变化的页面采用SSG静态生成，预渲染为静态HTML，访问速度极快。
2.  **渲染性能优化**
    - 虚拟列表：大数据量列表采用虚拟列表，仅渲染可视区域内的课程卡片，大幅减少DOM节点数量，降低重排重绘耗时，提升滚动流畅度；
    - 组件渲染优化：拆分频繁更新的组件，使用缓存组件、渲染优化API，减少不必要的组件重渲染；
    - 避免主线程阻塞：耗时的筛选、排序逻辑放入Web Worker执行，不占用浏览器主线程，避免页面卡顿。
3.  **体验兜底优化**：首屏加载时展示骨架屏，替代白屏；优先渲染导航、筛选栏、首屏列表等核心内容，非核心内容延迟渲染；弱网环境优先展示缓存历史数据，提升用户等待体验。

### 7.4 核心性能指标体系

采用Google Web Vitals核心指标+业务自定义指标，全面衡量优化效果：

1.  **LCP（最大内容绘制）**：核心指标，衡量首屏加载速度，指页面最大内容元素渲染完成的时间，优秀标准≤2.5s；
2.  **FCP（首次内容绘制）**：衡量白屏时长，浏览器首次渲染DOM内容的时间，优秀标准≤1.8s；
3.  **TTI（可交互时间）**：页面完全可交互的时间，无长任务阻塞主线程，优秀标准≤3.8s；
4.  **INP（交互延迟）**：衡量页面交互响应速度，用户交互到页面响应渲染的时间，优秀标准≤200ms；
5.  **CLS（累积布局偏移）**：衡量页面视觉稳定性，元素意外偏移的总量，优秀标准≤0.1；
6.  **业务自定义指标**：首屏接口请求耗时、列表渲染完成时间、筛选响应时间、页面滚动帧率。

## 八、常见标准状态码

### 1xx 信息性状态码
表示请求已接收，继续处理。
    100 Continue：客户端应继续发送请求体。
    101 Switching Protocols：服务器同意切换协议（如升级到 WebSocket）。

### 2xx 成功状态码
表示请求被成功处理。
    200 OK：请求成功，返回所需数据。
    201 Created：请求成功并创建了新资源（如 POST 请求后）。
    204 No Content：请求成功，但响应无内容（如 DELETE 请求后）。

### 3xx 重定向状态码
表示需要进一步操作以完成请求。
    301 Moved Permanently：资源已永久移动到新 URL。
    302 Found：资源临时重定向到新 URL。
    304 Not Modified：资源未修改，可使用客户端缓存（用于缓存验证）。

### 4xx 客户端错误状态码
表示客户端请求有误。
    400 Bad Request：请求语法错误，服务器无法理解。
    401 Unauthorized：请求需要身份验证（未提供或无效）。
    403 Forbidden：服务器理解请求，但拒绝执行（权限不足）。
    404 Not Found：请求的资源不存在。
    405 Method Not Allowed：请求方法（如 GET/POST）不被允许。
    429 Too Many Requests：客户端发送过多请求（限流）。

### 5xx 服务器错误状态码
表示服务器处理请求时出错。
    500 Internal Server Error：服务器内部错误，无法完成请求。
    502 Bad Gateway：作为网关或代理的服务器从上游服务器收到无效响应。
    503 Service Unavailable：服务器暂时无法处理请求（如过载或维护）。
    504 Gateway Timeout：网关或代理服务器未能及时从上游服务器收到响应。

### 其他常见网络相关状态码
    TCP/UDP 层错误：如连接超时、连接拒绝（非 HTTP 标准，但常见于网络故障）。
    DNS 错误：如域名解析失败。

### 实用场景示例
    访问网页时显示 “404”：页面不存在或 URL 错误
    提交表单后显示 “500”：服务器代码出错
    跳转时显示 “301”：网址已永久迁移，浏览器会自动跳转
