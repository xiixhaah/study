### 模板

在用户管理系统中，包含了用户、角色、部门等多核心管理模块，随着业务迭代模块持续增加，不同角色对应的页面访问权限、菜单可见性、按钮操作权限出现判断逻辑不统一、维护成本高的问题，甚至存在权限判断遗漏导致的越权访问风险。
作为核心开发人员，我负责该系统 RBAC（基于角色的访问控制）模型的前端开发，统一全系统权限判断逻辑并降低维护成本。
① 基于 Pinia 设计全局权限状态管理方案：将登录用户的角色信息、权限标识符列表、已授权路由列表存储在 Pinia 中，结合 localStorage 做持久化处理，解决页面刷新后权限状态丢失问题；封装通用权限判断函数，统一处理路由显示、按钮显隐、接口数据过滤的逻辑；
② 路由守卫统一权限拦截：在全局路由守卫中集成权限校验逻辑，用户跳转页面时匹配其权限标识与路由配置，未授权则拦截并跳转至无权限页面。
这套权限方案统一了全系统的权限判断逻辑，消除越权访问风险；后续新增模块仅需配置权限标识即可完成管控，模块接入成本降低，系统可维护性大幅提升。

## 亮点2：自定义hooks 抽留重复逻辑

**S（背景）**：用户管理系统的角色、部门、用户列表等模块，均涉及分页查询、条件搜索、数据刷新等重复业务逻辑，分散在各组件中导致代码冗余率高，一处逻辑修改需同步调整多个组件，易出错且维护效率低。

**T（任务）**：作为核心开发人员，我负责抽离通用业务逻辑，让页面聚焦于业务的展示，提升代码复用性与可维护性，减少重复开发成本。

**A（行动）**：
① 梳理高频重复逻辑：提炼分页、搜索等通用逻辑；
② 封装自定义 Hooks：开发 usePagination、useSearch等Hooks，将通用逻辑抽离为独立函数，通过参数传入模块专属配置；
③ 组件集成复用：各业务模块仅需引入 Hooks 并传入差异化配置，即可复用通用逻辑。
防止过度封装：并没有将所有通用状态抽离，业务强相关逻辑还是保留在当前页面，可以避免过度封装，使hooks达到效果。

**R（结果）**：单模块代码量减少，代码冗余率显著降低；后续逻辑迭代仅需修改 Hooks 内部，维护效率提升，避免多组件修改导致的逻辑不一致问题。 整体团队效率提升10%

## 亮点3：课程平台中通过组件化和筛选交互的设计来提升页面组织能力 构建层 架构

**S（背景）**：核心页面不仅有课程展示、课程筛选等内容，所有的功能堆在一个组件里，会造成状态的分散，渲染开销增加

**T（任务）**：针对问题做了一个课程筛选的页面，使页面结构划分，交互流畅

**A（行动）**：1、封装了axios请求，统一处理登录态以及接口的交互，为了让页面层不去耦合，我对axios进行了封装，包括状态码、错误处理等；2、对筛选条件之间的表单进行了联动处理；

**R（结果）**：思考请求逻辑的统一管理，状态应该放在哪一层，组件的边界怎么划分

在课程平台开发中，核心页面因课程展示、筛选等功能集中于单一组件，导致状态分散且渲染开销增加。针对这一问题，我们的任务是构建独立的课程筛选页面，以优化页面结构划分与交互流畅度。为此，一方面封装了 axios 请求，统一处理登录态及接口交互（含状态码、错误处理等），实现页面层解耦；另一方面对筛选条件间的表单进行联动处理。通过这些行动，我们进一步思考了请求逻辑的统一管理、状态的层级放置以及组件边界的划分问题。
